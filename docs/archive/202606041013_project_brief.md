## 1. Resumen ejecutivo

**ExportNotion** es una webapp Next.js 16 (App Router) desplegada en **Vercel Hobby** que permite descargar **CSV bajo demanda** (filtrado por rango de fechas) con los datos de una base de Notion. El export **nunca consulta Notion en vivo**: sirve desde un **snapshot cacheado en Upstash Redis**, que se rellena con syncs programados (crons) o manuales.

```
Notion ──(cron/manual sync)──► Upstash hash `notion:cache:v1` ──(GET /api/export)──► CSV stream
```

La decisión arquitectónica dominante es **sobrevivir los límites de plataforma**: Vercel Hobby mata funciones a los 60s y Notion corta cualquier query a 10,000 resultados. De ahí nace el **full sync chunkado por segmentos** con pivote reanudable — la pieza de mayor complejidad del sistema.

- **Usuario objetivo**: un equipo pequeño con un password compartido (auth de password único + sesión en cookie).
- **Madurez**: en producción (deploy en Vercel documentado), con manual de usuario y suite de tests.

## 4. Flujos de negocio

### 4.1 Full sync chunkado (el flujo crítico)
1. Cron 09:00 UTC o botón "Full" → `POST /api/sync?kind=full` (adquiere lock).
2. Primer segmento: limpia `notion:cache:v1:new`, abre session flag.
3. `fetchOneFullSegment` trae páginas DESC por `created_time` con presupuesto de 25s → flatten → upsert al hash `:new`.
4. Si quedan datos: guarda pivote, responde `done:false`; **el cliente vuelve a llamar** (encadena hasta 20 segmentos).
5. Al completar: RENAME atómico `:new` → cache vivo, actualiza meta, limpia pivote/session.
⚠️ **El cron solo dispara UN segmento**: con >10k filas, el full nocturno no termina solo — alguien debe abrir la UI.

### 4.2 Incremental
Cron 21:00 UTC o botón → query con filtro `last_edited_time > lastIncrementalAt − 60s` → upsert + delete de archivadas. Una pasada, siempre `done:true`.

### 4.3 Export CSV
Login → `GET /api/export?from&to` → lee todo el hash (HSCAN), filtra por `DATE_COLUMN`, ordena asc en memoria, stream CSV con BOM (nombre `export-{from}-{to}-{timestamp}.csv`).

## 8. Lagunas (qué falta saber / deuda conocida)

1. **Operación del full >10k depende de un humano** — el cron no encadena segmentos (mitigable con segundo cron, QStash o plan Pro).
2. **Dos archivos de sesión** (`src/lib/auth.ts` y `src/lib/session.ts`) — middleware importa de uno, routes del otro; consolidación pendiente.
3. **Spec y plan originales desactualizados** vs implementación post-MVP (plan truncado: tasks 9–22 sin escribir; ejecutadas según git log).
4. **Sin guía de migración de `COLUMNS`/`DATE_COLUMN`** post-deploy (¿se invalida el cache?, ¿cambia la estructura del CSV?).
5. **Export carga todo el dataset en memoria** para filtrar/ordenar — aceptable a decenas de miles de filas, límite conocido.
6. **`COLUMNS` hard-codeada en español** — el setup por proyecto exige editar código.
7. **Next 16 sugiere `middleware.ts` → `proxy.ts`** — pendiente.
8. **E2E falla en local sin `UPSTASH_*` reales** (`Redis.fromEnv()` revienta en la primera request).
